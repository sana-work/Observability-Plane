Architecture

End-to-End Platform Architecture

Full view of the AI Services Observability Plane — from service instrumentation through Kafka event processing, multi-layer storage, to the observability dashboards.

Data Storage

Multi-Layer Storage Strategy

Five purpose-built storage layers — each optimised for a distinct access pattern. Snowflake and Redis are under active evaluation; PostgreSQL and an in-process cache serve as production-ready interim solutions until onboarding is confirmed.

🔎  Snowflake & Redis are under evaluation — onboarding is being discussed but not yet approved. The current interim (PostgreSQL obs_events.* + in-process TTL cache) is production-ready and the swap path is contained.
🔍
Elasticsearch — Hot Operational
Elasticsearch / Kibana

Per-LOB namespaced indices for requests, errors, agent steps, LLM calls, tool calls, RAG events, guardrail decisions, feedback, traces, anomalies, quality scores. Sub-second full-text search. ILM: 30-day default, 180-day for regulated LOBs.

🗂️
PostgreSQL — Control Plane Only
PostgreSQL observability.*

Registries (app/agent/tool/rag/prompt), KPI & metric catalog, budget limits, daily SLO compliance, alert thresholds, dashboard config, feedback cases. Transactional integrity for small authoritative config — not a firehose sink.

📊
PostgreSQL — Event Firehose (Interim)
PostgreSQL obs_events.*

Interim while Snowflake evaluation is ongoing. Monthly partitioned tables for all enriched events with JSONB payload. ~90-day hot window; old partitions archived to S3. Swap to Snowflake if/when approved — identical event_id/correlation_id model.

☁️
Snowflake — Analytics / Forever
Snowflake · Under Evaluation

Target analytics store: all events, forever retention, on-demand heavy queries for chatbot and trend dashboards. No pre-aggregation. Per-LOB RBAC. Onboarding in discussion — PostgreSQL obs_events.* bridges the gap until confirmed.

📦
Amazon S3 — Payload Archive
Amazon S3

Redacted prompts/responses, full trace JSON, RAG contexts, audit evidence, multimodal document artifacts, debug bundles, RCA reports, IaC dashboards. S3 Standard → IA @ 30d → Glacier @ 180d. SSE-KMS encrypted.

⚡
Redis — Runtime State
Redis / ElastiCache · Under Evaluation

Target: sub-ms budget accumulator (INCRBYFLOAT), shared registry cache (cross-pod), dedup fast-path (SETNX), rate limits, active correlation context. Onboarding in discussion — interim: Postgres budget_accumulator + in-process TTL cache.

Event Pipeline

Centralised Event Processing

Every event from all 8 services passes through a single 9-stage enrichment pipeline before reaching any storage. PII redaction, cost calculation, SLO evaluation, and quality flagging are handled centrally — services are never burdened with this cross-cutting work.

09

Flag RAG and LLM events for sampling by the quality evaluation service
🔒

Two-Layer Architecture

Layer 1 — AI Quality & Trace: LLM traces, RAG spans, agent step trees, prompt versioning, faithfulness evaluations, and feedback linkage — targeted at product and AI quality teams.
Layer 2 — Platform & Infrastructure: Structured events, Kafka health, cost governance, SLO burn rates, business KPIs, and anomaly detection — targeted at platform engineering and management.
Both layers share a single correlation_id — any request is reconstructable across both.

Monitoring · Infrastructure Layer

Infrastructure Observability

Grafana Tempo provides distributed tracing; kube-prometheus-stack delivers infrastructure metrics — both feed the custom dashboard backend directly. Grafana's own visualisation layer is intentionally disabled: the platform requires COIN-JWT auth, per-LOB RBAC, and business KPI views that Grafana cannot model cleanly.

⚠️

Deliberate Design Decision — Grafana Visualisation Disabled

kube-prometheus-stack is deployed with grafana.enabled=false. Prometheus metrics are consumed directly by the Custom Dashboard Service via PromQL. This gives us COIN-JWT SSO, per-LOB row-level access control, and business KPI views natively — capabilities that would require significant workarounds in Grafana.

🔭
Grafana Tempo — Distributed Traces
Grafana Tempo · OTEL → S3 backend

Receives OTEL spans from all 8 services via auto-instrumentation (FastAPI/httpx/asyncpg). S3 backend at ~$5/mo per million spans/day. Cross-service span view via TraceQL. Keyed by the same correlation_id as the event pipeline.
